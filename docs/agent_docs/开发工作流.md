# 开发工作流

> 本文件是 CLAUDE.md 的外链细则，描述开发一个新功能的**完整七步流程**。
> 触发条件：开始一个新功能、修复 bug、重构现有代码。

---

## 第一步：需求讨论
- 王总提出创意和想法
- 我用通俗语言分析可行性和实现思路
- 确认需求后进入下一步

## 第二步：规划方案
- 对于复杂功能，我会制定实施计划（使用 `superpowers:writing-plans` 或 `ecc:planner`）
- 涉及系统架构决策时，会使用 `ecc:architect` 进行架构设计
- 与王总确认计划后再开始编码

## 第三步：编码实现
- 我会完成全部 C# 脚本编写
- 代码遵循以下规范：
  - 使用 PascalCase 命名类和方法
  - 使用 camelCase 命名变量和参数
  - 私有字段加 `_` 前缀（如 `_health`）
  - 每个方法不超过 50 行
  - 添加必要的中文注释方便王总理解
- 实现方式视场景选择：
  - 一般功能：直接编写
  - 复杂功能：使用 `superpowers:subagent-driven-development` 分步实现
  - 需要重构优化时：使用 `ecc:refactor-clean` 清理代码

## 第四步：质量保障
- **代码审查** — 使用 `ecc:code-review` 或 `superpowers:requesting-code-review` 进行审查
- **安全审查** — 涉及用户输入、网络通信等功能时使用 `ecc:security-review`
- **隐蔽 Bug 排查** — 使用 `ecc:silent-failure-hunter` 检查沉默错误
- **性能优化** — 性能敏感场景使用 `ecc:performance-optimizer`
- **构建错误** — 如 Unity 构建失败，使用 `ecc:build-error-resolver` 排查
- **最终验证** — 使用 `verification-loop` 或 `superpowers:verification-before-completion`
- 我会提供详细的测试操作步骤给王总在 Unity Editor 中执行

## 第五步：更新进度文档
- 更新 `docs/项目进度.md`，把刚完成的功能标记为「已完成」
- 记录完成日期和必要的说明

## 第六步：提交代码
- 遵循 ECC git-workflow 规范
- 提交格式：`<type>: <描述>`
- 类型包括：feat（新功能）、fix（修复）、refactor（重构）、docs（文档）
- 提交到 GitHub 远端仓库

## 第七步：会话管理
- 每次完成一个功能并提交代码后，我会建议王总清空对话
- **为什么要清**：聊得越久我越容易"忘事"，就像草稿纸写满了要换新的
- **清空对话很简单**：
  1. 输入 `/clear` 然后按回车，对话就清空了
  2. 然后跟我说「看看进度」，我读完进度文档就什么都知道了

---

## 开发相关的额外注意事项

1. **本项目所有代码由 AI（我）生成**，需要经过王总在 Unity 中验证
2. **我不是 Unity 引擎**，无法直接操作 Unity 编辑器 — 所有引擎操作需要王总在 Unity 中执行
3. **每次完成一个功能后，必须更新 `docs/项目进度.md`**，包括：
   - 把对应的「计划中」或「正在做」改为「已完成」
   - 记录完成日期
   - 如果有重要的技术决策，同步更新「决策记录」部分
   - 让王总能随时看到项目的最新状态
4. 始终保持项目文件结构的整洁，避免冗余文件和混乱的资源管理
5. **关于"聊久了会忘事"**：随着对话变长，我会逐渐"忘记"前面的内容。这不是出问题了，是正常现象。如果王总发现我反应变慢或总忘事，跟我说「保存一下」，我提交代码更新进度。然后王总输入 `/clear` 回车清空对话，再跟我说「看看进度」，我读了进度文档就什么都知道了。
